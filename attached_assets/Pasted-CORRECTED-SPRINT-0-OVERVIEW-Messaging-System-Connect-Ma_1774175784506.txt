IMPORTANT NOTES
Regarding existing tables (FIXES #19, A): Your app already has group_messages,
grow_group_messages, channels, and channel_messages tables. The new messaging tables are
COMPLETELY SEPARATE. Do NOT modify the existing tables. The new messaging channels table is
named 'connect_channels' (not 'channels') to avoid conflicts. Messages are already named
'connect_messages'.
Regarding workspaces (FIXES #10, B, D): Your app DOES have a workspace system (workspaces table with
serial/integer PK, workspace_members, teams). All workspaceId columns in the new messaging schema use integer
type to match the existing workspaces.id. Use the existing workspace system.
Regarding user IDs (FIX C): Your users table uses varchar('id') for Replit Auth sub values (e.g. '12345').
ALL user-referencing columns in the new schema (authorId, createdBy, userId, pinnedBy, startedBy)
use text/varchar — NOT uuid. This matches req.user.claims.sub.
Regarding Redis (FIXES #11, #15): All services are written with in-memory fallbacks. Redis is optional and can be
added later for production scaling.
Regarding external services (FIXES #12, #13, #16): File uploads via Cloudflare R2, voice huddles via
LiveKit/Daily.co, and push notifications via Firebase are all deferred to future sprints. Initial build uses local
alternatives or omits these features.
